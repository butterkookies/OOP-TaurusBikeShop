## SECTION 1 — WHAT THIS PROJECT IS

Taurus Bike Shop is a full-stack bike shop management system with two applications
sharing one database (TaurusBikeShopDB on Google Cloud SQL for SQL Server):

- **WebApplication** — ASP.NET Core MVC (.NET 8), customer-facing: product browsing,
  cart, checkout, payment, order tracking, reviews, wishlist, support ticket creation.
- **AdminSystem** — WPF (.NET Framework 4.8), staff-facing: POS, inventory, order
  management, logistics, payment verification, reports, vouchers, support,
  supplier/purchase orders.

They share ONE database only — no shared code, no shared assemblies.

## SECTION 2 — TECHNOLOGY STACK

- **WebApplication:** ASP.NET Core MVC (.NET 8), EF Core, BCrypt.Net-Next,
  Google.Cloud.Storage.V1, Bootstrap (from ~/lib/bootstrap/), jQuery, jQuery Validation
- **AdminSystem:** WPF (.NET Framework 4.8), Dapper, ADO.NET, BCrypt.Net-Next,

- No Entity Framework in AdminSystem — Dapper + raw SQL only
- No Razor Pages anywhere — MVC controllers and views only in WebApplication

## SECTION 3 — ABSOLUTE BUSINESS RULES — NEVER VIOLATE

1. **No COD** — PaymentMethod values: GCash, BankTransfer, Cash (POS only). No card, no COD.
2. **No refunds** — Taurus does not offer refunds. No code path ever sets PaymentStatus = 'Refunded'.
3. **Couriers: Lalamove and LBC only** — no other courier values anywhere in code.
4. **Stock lives exclusively in ProductVariant.StockQuantity** — no StockQuantity on Product table.
5. **Every Product must have at least one ProductVariant row** — use VariantName = 'Default' for products with no real variants.
6. **Computed fields are never stored** — TotalAmount (Order), Subtotal (OrderItem),
   Subtotal (PurchaseOrderItem), TotalAmount (PurchaseOrder) are all computed in views.
7. **5 shared-PK 1:1 relationships** — GCashPayment, BankTransferPayment, LalamoveDelivery,
   LBCDelivery, PickupOrder all share PK with their parent table. PKs are ValueGeneratedNever().
8. **Circular FK** — User.DefaultAddressId → Address.AddressId is deferred FK,
   configured in AppDbContext with IsRequired(false) and OnDelete(DeleteBehavior.NoAction).
9. **IssueRefund task type was removed** — SupportTask.TaskType valid values:
   ShipReplacement, ArrangeReturn, ContactSupplier, Other. Never add IssueRefund back.
10. **InventoryLog rows are immutable** — append-only, never update or delete.
11. **Address snapshot rows (IsSnapshot=1) are immutable** — never modify after creation.
12. **PriceHistory rows are immutable** — written only by DB trigger TR_Product_PriceAudit.
13. **Order+OrderItem+InventoryLog** must always be written in a single DB transaction.

## SECTION 4 — ARCHITECTURE STANDARDS (APPLY TO EVERY FILE)

### Naming
- Interfaces: IServiceName, IRepositoryName
- Async methods: always suffix with Async
- Private fields: _camelCase
- ViewModels: suffix ViewModel
- DTOs/results: suffix Result or Dto
- EF entities: match DB table name exactly
- Controllers: suffix Controller
- Repositories: suffix Repository
- Services: suffix Service

### Architecture Rules
- Controllers are dumb — receive request, call one service method, return result.
  No business logic, no direct DB access in controllers.
- Services own all business logic — validation, rules, status transitions,
  notification queuing. Services call repositories, never DbContext directly.
- Repositories own all data access — no LINQ in controllers, no raw SQL in services.
  Never expose IQueryable outside repository boundary.
- Interfaces for everything injectable.
- EF Core configuration in OnModelCreating ONLY — no relationship data annotations
  on entity classes. Data annotations only for validation (Required, MaxLength).
- All AJAX controller actions return ApiResponse envelope:
  public class ApiResponse { public bool Success; public string? Message; public object? Data; }

### .NET Standards
- async/await throughout — every method touching I/O must be async
- CancellationToken in background jobs and long-running repository calls
- IReadOnlyList<T> or IEnumerable<T> for collection returns — never List<T> as public contract
- Nullable reference types enabled — every nullable property marked with ?
- Never use var where type is not immediately obvious
- Use record types for immutable DTOs and result objects
- Use sealed on classes not designed for inheritance
- String constants for status values — never magic strings
- Guard clauses at top of methods — fail fast, return early
- All public methods have XML doc comments (/// <summary>)

### Security
- BCrypt work factor 12. Never log/serialize/transmit password hashes.
- Zero tolerance SQL injection — EF Core parameterized queries in WebApplication
- [ValidateAntiForgeryToken] on every POST action
- [Authorize] on every action requiring authentication
- File uploads: validate MIME type and size before GCS upload. Max 5MB.

### Transaction Standards
- Any operation writing to more than one table = single DB transaction. No exceptions.
- Critical transactions: CreateOrder, CreateWalkInOrder, ReceiveStock, VerifyPayment
  must explicitly rollback on failure and log to SystemLog.

### Background Job Standards
- All 5 jobs extend BackgroundService, implement ExecuteAsync(CancellationToken)
- Every cycle writes BackgroundJobStart and BackgroundJobComplete/Error to SystemLog
- Jobs handle external API failures gracefully — catch, log, continue next cycle
- Jobs avoid duplicate notifications — check Notification table before queuing
- Use IServiceScopeFactory to resolve scoped services inside singleton BackgroundService

Define generic interface IRepository<T> where T : class

- Task<IReadOnlyList<T>> GetAllAsync(CancellationToken cancellationToken = default)
- Task<T?> GetByIdAsync(int id, CancellationToken cancellationToken = default)
- Task AddAsync(T entity, CancellationToken cancellationToken = default)
- Task UpdateAsync(T entity, CancellationToken cancellationToken = default)
- Task DeleteAsync(int id, CancellationToken cancellationToken = default)
- Task<IReadOnlyList<T>> FindAsync(Expression<Func<T, bool>> predicate, CancellationToken cancellationToken = default)

### Repository.cs
Location: WebApplication/DataAccess/Repositories/Repository.cs
Namespace: WebApplication.DataAccess.Repositories

Implement IRepository<T> using AppDbContext.
- Constructor takes AppDbContext via DI
- GetAll: DbSet<T>.AsNoTracking().ToListAsync()
- GetById: DbSet<T>.FindAsync(id)
- Add: DbSet<T>.AddAsync + SaveChangesAsync
- Update: DbSet<T>.Update + SaveChangesAsync
- Delete: find by id, remove, SaveChangesAsync — throw if not found
- Find: DbSet<T>.AsNoTracking().Where(predicate).ToListAsync()
- Protected property DbSet<T> DbSet for subclasses to use
- Protected AppDbContext Context for subclasses that need transaction access
